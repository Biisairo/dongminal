# dongminal-team

dongminal MCP 를 활용해 **여러 Claude Code 인스턴스가 같은 워크스페이스 안에서 서로 메시지를 주고받으며 협업**하도록 만드는 범용 스킬. 기존 Claude Code 의 tmux 기반 team agents 가 tmux 의존 때문에 쓰기 어려웠던 문제를, dongminal 의 pane/세션/탭 + 신뢰 채널(`send_agent_message`) 로 해결한다.

이 스킬은 특정 시나리오(예: 코드 리뷰 팀, GAN 팀)를 하드코딩하지 않는다. **팀을 조립하고 · 작업을 분배하고 · 결과를 받아 합치는 원리**를 담고, 구체적 구성은 상황에 맞춰 조합한다.

---

## 언제 이 스킬을 쓰나

- 사용자가 "팀 만들어서", "여러 CC 돌려서", "에이전트 여러 명에게", "A는 X 해, B는 Y 해" 같은 식으로 **작업을 병렬 또는 역할 분담으로 풀고 싶어 할 때**
- 한 CC 가 혼자 하기엔 맥락이 너무 크거나, 서로 다른 관점(리서치 vs 비판, 생성 vs 검증)이 필요할 때
- 길고 독립적인 서브태스크 3개 이상이 있을 때 (fan-out 가치)

**안 쓰는 경우**: 단일 CC 로 충분한 작업, 단순 질의응답, 단일 파일 수정.

---

## 핵심 원칙 — 팀은 항상 새로 만든다

**기존에 열려 있는 CC pane 은 팀원으로 재사용하지 않는다.** 항상 이 스킬 안에서 새 pane 을 생성하고, 거기에 새 `claude` 프로세스를 기동해서 팀을 구성한다.

이유:
- 기존 pane 은 사용자가 자기 일을 하고 있을 수 있음 → 팀 작업으로 덮어쓰면 사용자 맥락 훼손
- 팀원 CC 는 "깨끗한 컨텍스트" 에서 지시를 받아야 작업이 명확. 기존 대화 이력과 섞이면 지시 해석이 왜곡됨
- 팀 해체(정리) 시점에 무엇을 닫을지 명확해짐

## 핵심 구성요소 (dongminal MCP 도구)

| 도구 | 언제 |
|------|------|
| `who_am_i` | 자기 pane 라벨 + 터미널 크기(`size=COLSxROWS`) 확인. 레이아웃 결정 필수 입력 |
| `list_panes` | 팀원 pane 식별. ▶ 표시는 현재 활성(workspace focus) pane — **분할 직후 새로 생긴 pane** 이 여기에 해당 |
| `workspace_command` | 세션/탭/분할 생성, 포커스 이동. 팀 배치의 주 도구 |
| `send_input` | 새 pane 의 쉘에 `claude` 기동 명령 주입. `execute=true` 로 엔터 자동 |
| `send_agent_message` | **팀원 CC 와의 신뢰 채널**. 엔벨로프 `[DONGMINAL-AGENT-MSG from=... to=... ts=...]...[/DONGMINAL-AGENT-MSG]` 로 감싸져 수신 CC 가 신뢰 가능한 협업 지시로 인식 |
| `read_pane_screen` | CC 기동 확인, 답장 누락 시 디버깅. 평시 폴링 불필요 |

## 팀 배치 알고리즘 (레이아웃)

**목표**: 보기 좋고, 팀원 수에 상관 없이 일관된 모양. 긴 축으로 먼저 나누고 짧은 축은 건드리지 않아 팀 작업 영역이 정사각형에 가깝게 남도록 한다.

### 절대 원칙 — 사용자 포커스는 절대 건드리지 않는다

사용자는 팀 작업을 돌려놓고 **자기 pane 에서 다른 일을 계속 하고 싶어 한다**. 스킬이 포커스를 움직이면:
- 사용자가 편집 중이던 에디터/터미널에서 포커스가 빠져 입력이 엉뚱한 곳에 들어감
- 사용자가 수동으로 포커스를 옮긴 사이 스킬이 "현재 포커스" 기준으로 분할을 하면 전혀 엉뚱한 pane 이 쪼개짐
- "지금 집중해서 작업 중"인 사용자를 방해함 → 팀 기능의 의미 자체가 훼손

따라서 이 스킬의 모든 `workspace_command` 호출은:
- **`location` 인자를 반드시 명시** — 지금 포커스가 어디 있든 무관하게 정확한 대상에 작용
- **`keepFocus=true` 를 항상 붙임** — `splitH/splitV` 에서 포커스 유지. `location` 지정 `closeTab` 은 서버가 `keepFocus` 여부와 무관하게 포커스를 건드리지 않는다. 따라서 분할·해체 모두 포커스가 1프레임도 움직이지 않는다.
- **`focus` 액션은 스킬 내에서 절대 사용 금지.** 어떤 상황에서도. 복원 목적도 포함. 이유: 사용자가 현재 다른 pane 에서 입력 중이면 포커스 이동 자체가 방해. "원래 자리로 돌려놓는다" 는 의도도, 사용자가 그 사이에 다른 곳으로 이동했을 수 있으므로 결과적으로 엉뚱한 곳으로 포커스를 보냄. 스킬은 사용자가 지금 어디에서 무얼 하고 있는지 알 수 없으므로, 포커스 제어 권한을 가지면 안 된다.
- `list_panes` / `read_pane_screen` / `send_input` / `send_agent_message` 는 전부 명시적 라벨 기반이라 포커스와 무관 — 그대로 사용. 팀 작업 전 과정이 사용자 포커스와 분리되어 진행된다.

### 단계

1. `who_am_i` → 팀장 라벨 `BOSS` (예: `S4.P3.T1`) + `size=COLSxROWS` 획득 (셀 단위). 이 라벨은 이후 분할의 `location` 으로만 쓰며, 포커스 이동에는 쓰지 않는다.
2. **시각적 긴 축 방향을 결정** (셀 비율 보정 필수):
   - 터미널 셀은 정사각형이 아니다. 한 셀의 높이는 너비의 약 **2.2배** (사용자 환경 캘리브레이션 값; 폰트/라인스페이싱에 따라 2.0~2.5 범위에서 변동)
   - 그래서 `COLS`/`ROWS` 숫자 그대로 비교하면 시각과 어긋남. 시각적 픽셀 비율은 대략 `COLS : ROWS × 2.2`
   - **판정 규칙**: `COLS >= ROWS * 2.2` → 시각적으로 가로가 긴 것 → 1차 분할 `splitH` (좌↔우). 팀 영역은 오른쪽
   - 반대(`COLS < ROWS * 2.2`) → 시각적으로 세로가 긴 것 → 1차 분할 `splitV` (상↕하). 팀 영역은 아래쪽
   - 임계값이 애매한 경계 케이스에서는 `splitV` 쪽으로 치우치는 게 (가로 공간이 충분히 남아 각 팀원 셀폭이 여유 있어) 보수적으로 안전
3. **1차 분할 실행**: `workspace_command(<splitH|splitV>, location=BOSS, keepFocus=true)`
   - `location=BOSS` 로 팀장 pane 을 쪼갠다 (현재 포커스가 어디에 있든 무관).
   - `keepFocus=true` 로 분할 후에도 사용자 포커스는 그대로. 사용자는 자기가 보던 곳에서 계속 작업.
   - 이 시점에 팀 영역에는 **팀원 pane 1개** 가 이미 존재 (N=1인 팀이면 여기서 종료 — 해당 pane 라벨을 `list_panes` 로 확인)
4. **팀원 수에 맞춰 직교 축으로 한 번에 N 분할**:
   - 1차가 `splitH` 였다면 직교 축은 `splitV`, 반대도 마찬가지
   - 먼저 `list_panes` 를 호출해서 **1차 분할로 새로 생긴 팀 영역 pane 라벨 `SEED`** 를 확인한다 (팀장과 같은 세션 안의 새 P 번호)
   - N≥2 이면 `workspace_command(<splitV|splitH>, location=SEED, count=N, keepFocus=true)` **단일 호출**로 팀 영역을 N 균등 분할
   - 내부적으로 (N−1) 개 pane 이 생성되고 `SEED` + 신규들이 같은 split 노드의 형제로 배치되어 시각적으로 정확히 N 등분됨. `count` 미지정이면 기본 2 분할
   - `keepFocus=true` 덕에 분할 후에도 사용자 포커스는 움직이지 않음. 팀원 pane 들의 positional 라벨은 다음 단계의 `list_panes` 로 일괄 확보
5. `list_panes` 로 팀원 pane 라벨들을 전부 수집. **CC 기동은 아직 하지 않는다** — 분할이 모두 끝났으므로 다음 단계에서 일괄 기동

### 이유

- **긴 축 기준 분할**: 만약 짧은 축을 먼저 나누면, 팀 영역이 너무 납작해져 각 팀원 터미널이 사용 불가능한 크기가 됨. 긴 축을 먼저 나누면 팀 영역이 충분한 공간을 확보한 상태로 내부 분할이 진행된다.
- **직교 축 재분할**: 팀 영역 안에서 1차와 같은 축으로 또 나누면 점점 더 납작해진다. 직교로 나눠야 각 팀원이 대략 비슷한 비율을 유지한다.
- **한 번에 N 분할**: 과거에는 (N−1) 번 반복 호출해 이진 분할을 중첩시켰고 결과 사이즈가 50%/25%/25% 처럼 불균등해졌다. 이제 `count=N` 한 번으로 정확히 균등 N 등분된다. 왕복 횟수도 줄어 레이스 표면적이 작아진다.

### 예시

**예: COLS=200, ROWS=50, 팀원 3명, 팀장 `S4.P3.T1`**
- `50 * 2.2 = 110`. `200 >= 110` → 1차 `splitH`
- `workspace_command(splitH, location="S4.P3.T1", keepFocus=true)` → `list_panes` 로 SEED 확인 (예: `S4.P4.T1`)
- `workspace_command(splitV, location="S4.P4.T1", count=3, keepFocus=true)`
- 결과: 왼쪽 boss 영역, 오른쪽 팀 영역이 team1/2/3 세로 3 등분. 사용자 포커스는 분할 내내 고정.

**예: COLS=132, ROWS=63, 팀원 2명, 팀장 `S4.P3.T1`**
- `63 * 2.2 = 138.6`. `132 < 138.6` → 1차 `splitV`
- `workspace_command(splitV, location="S4.P3.T1", keepFocus=true)` → SEED 확인
- `workspace_command(splitH, location="<SEED>", count=2, keepFocus=true)`

**예: COLS=80, ROWS=80, 팀원 4명** — `80 < 176` → 1차 `splitV` → 직교 `splitH` with `count=4`, 둘 다 `location=<대상>` + `keepFocus=true`.

## 팀원 CC 기동 — 3단계 (병렬 부팅 → Barrier → Kickoff)

팀원 CC 기동은 **반드시 아래 3단계**로 분리한다. 이 순서가 깨지면 데드락이 발생한다.

1. **병렬 부팅** — 모든 팀원에 `claude --model X "<대기용 프롬프트>"` 를 한 메시지에서 병렬 `send_input`. 프롬프트에는 역할·팀 구성·프로토콜·"대기하라"만 넣는다. **첫 작업 지시는 절대 포함하지 않는다.**
2. **Barrier 대기** — `read_pane_screen` 으로 모든 팀원 pane 에서 CC 입력 UI (`╭─`/`>` 프롬프트 박스) 가 확인될 때까지 반복. 전원 준비 완료 전까지 kickoff 안 함.
3. **Kickoff** — barrier 통과 후, 작업 개시자(들)에게 **`send_agent_message` 로** 첫 작업 지시를 보낸다. 이 시점엔 수신 팀원들이 전부 CC 상태이므로 수신자가 존재하지 않아 메시지가 증발하는 데드락이 나지 않는다.

### 왜 이렇게 분리하는가 — 데드락 실화

과거에는 초기 프롬프트에 "바로 작업 시작 + 동료 X 에게 결과 전송" 을 모두 inline 으로 넣었다. 문제:

- **CC 부팅 시간은 팀원마다 다르다.** opus vs sonnet, 네트워크/캐시 상태, terminal resize 등으로 수 초 단위 편차. 한 메시지에서 병렬 `send_input` 으로 동시에 기동해도 **CC 가 `>` 입력 UI 에 도달하는 시점은 다르다.**
- 먼저 부팅된 팀원(예: 작가) 이 inline 프롬프트를 받자마자 작업을 시작해 `send_agent_message` 로 결과를 송신한다. 그런데 **수신자(예: 수석 비평가) 는 아직 쉘 상태** → `send_agent_message` 가 쉘에 텍스트만 찍히고 엔터되어 명령 에러로 증발. 송신자는 "완료" 로 턴 종료, 수신자는 영원히 대기 → **데드락.**
- 팀 내 수평 통신이 있는 구조(작가→비평가, 비평가끼리 토론 등) 에서 거의 확정적으로 재현.

근본 원인: "pane 이 동시에 존재함" ≠ "CC 가 동시에 입력 받을 준비됨". Barrier 없이 kickoff 하면 안 된다.

**동시에 기동하는 의미는 유지한다** — 순차로 기동하면 먼저 뜬 팀원이 아직 만들어지지 않은 동료 라벨을 못 찾는 별개 문제가 있다. 병렬 기동으로 pane 들을 동시에 존재시키되, kickoff 은 barrier 뒤로 미룬다.

### 단계 1 — 병렬 부팅 (대기용 프롬프트)

각 팀원 pane 에 아래 형태로 주입:

```
claude --model <모델ID> "<역할·팀구성·프로토콜 + 마지막에 '[대기] 팀장의 kickoff 지시 도착 시까지 대기. 지금 아무 작업도 시작하지 말 것.'>"
```

팀원 수만큼 `send_input` 호출을 **단일 어시스턴트 메시지의 병렬 tool 호출**로 묶는다. 같은 메시지 안의 독립 tool call 은 병렬 실행되므로 N명이 사실상 동시에 부팅된다.

예시 (N=3 팀, 한 메시지에 세 호출):
```
send_input(id="S4.P4.T1", text="claude --model sonnet \"...역할A... [대기] ...\"", execute=true)
send_input(id="S4.P5.T1", text="claude --model haiku \"...역할B... [대기] ...\"", execute=true)
send_input(id="S4.P6.T1", text="claude --model haiku \"...역할C... [대기] ...\"", execute=true)
```

프롬프트에 포함시킬 것 / 빼야 할 것:
- **포함**: 역할, 팀 구성(모든 팀원 라벨 + 역할), 프로세스 흐름(전체 단계), 답장 포맷/규칙, 툴 풀네임, "대기" 지시
- **제외**: "지금 X 를 시작하라", "task-id=Y 로 첫 결과 송부" 같은 즉시 실행 지시

### 단계 2 — Barrier: 전원 CC 준비 완료 확인

`read_pane_screen` 으로 각 팀원 pane 을 확인. CC 입력 UI 의 특징:
- 화면 하단에 `╭─` 로 시작하는 테두리 박스와 `>` 프롬프트
- 혹은 "for shortcuts" / "Try \"...\"" 같은 CC 전용 안내 문구

모든 팀원이 **자기에게 주입된 대기용 프롬프트를 첫 유저 턴으로 처리 완료** 하고 입력 대기 상태로 돌아온 것을 확인해야 한다. 처리 중이면 "Thinking..." / 진행 바가 보인다 — 이 상태면 아직 준비 안 됨.

Barrier 알고리즘:
1. 병렬 부팅 직후 약 5~10초 기다린 뒤 (sleep 말고, 다른 할 일을 하거나 그냥 다음 tool 호출로 진입) 한 번에 모든 팀원 `read_pane_screen` (병렬 호출).
2. 전원 준비 완료면 단계 3 진입.
3. 준비 안 된 팀원이 있으면 해당 팀원만 다시 확인. 반복.
4. 30초 넘게 한 팀원이 준비 안 되면 실패로 판정 — 해당 pane 의 쉘 화면을 사용자에게 보여주고 진단(예: `claude: command not found`, opus 쿼터 초과 에러 등).

### 단계 3 — Kickoff (+ 제출 확인)

Barrier 통과 후, 작업 개시자(들) 에게 **`send_agent_message` 로** 첫 지시를 보낸다. 초기 프롬프트에서 이미 역할·프로토콜을 알고 있으므로 kickoff 메시지는 짧아도 된다. 예:

```
send_agent_message(
  to="S4.P4.T1",  # 작가
  from="<팀장라벨>",
  message="[TEAM-KICKOFF task-id=poem-draft-1]\nstatus: START\n시작하라. 주제: '봄의 기억'. 초안을 S4.P5.T1 에게 송부.\n[/TEAM-KICKOFF]"
)
```

Kickoff 송신 후 수신 pane 이 본문을 처리 시작했는지 `read_pane_screen` 으로 한 번 확인하면 안전하다 (`Thinking...` / 응답 텍스트 발생 = 정상). `send_agent_message` 는 paste → 지연 → `\r` 분리 전송으로 submit race 가 해소되어 있어 대부분 한 번에 제출되지만, 드물게 입력창에 `[Pasted text ...]` 가 남아 있으면 `send_input(id=<수신pane>, text="", execute=true)` 로 엔터 보강한다.

### 모델 선택 (`--model` 플래그)

팀원의 역할에 맞춰 모델을 고른다. 한 세션 전체를 Opus 로 돌리면 비싸고 느릴 수 있고, 반대로 창의성·복잡도가 필요한 역할에 Haiku 를 쓰면 품질이 떨어진다.

| 모델 인자 | 별칭 | 권장 역할 |
|-----------|------|-----------|
| `opus` 또는 `claude-opus-4-7` | Opus 4.7 | 창작 주연(시 쓰기, 서사 설계), 복잡한 합성·합의 도출, 아키텍처 설계 리더 |
| `sonnet` 또는 `claude-sonnet-4-6` | Sonnet 4.6 | 대부분의 리서치·코드 리뷰·중간 합성 — 기본 추천값 |
| `haiku` 또는 `claude-haiku-4-5` | Haiku 4.5 | 단순 분류/추출, 보조 비평, 짧은 검증 역할, 많은 수의 경량 워커 |

실제 사용 가능 별칭은 `claude --help` 또는 `claude --model` 오류 메시지로 확인 가능. 확실한 방법은 **full model ID** (`claude-opus-4-7` 등) 사용.

**기본 선택 가이드**:
- 팀원 역할이 명확히 다르면 각자 맞는 모델
- 전부 비슷한 수준의 리서치면 전원 Sonnet
- 팀장(본인) 은 현재 모델 유지 — 스킬이 바꿀 수 없는 영역

### 초기 프롬프트 안에 담아야 할 것

CLI 위치 인자로 넣는 프롬프트는 곧 팀원 CC 의 첫 사용자 메시지가 된다. 아래를 포함시키면 `send_agent_message` 로 따로 지시할 필요가 없다:

```
[역할] <한 줄 역할 + 이 팀원이 어떤 포지션인지>

[팀 구성]
  • 팀장:     <팀장 라벨>
  • 네 라벨:  <이 팀원 라벨>
  • 동료1:    <라벨>  (역할: ...)
  • 동료2:    <라벨>  (역할: ...)
  ...

[프로세스] (팀 내 통신 흐름이 있을 때)
  <누가 누구에게, 어떤 포맷으로, 몇 라운드에 걸쳐 대화하는지>

[작업] task-id=<id>
  <구체적 태스크>

[답장 규칙]
  반드시 **mcp__dongminal__send_agent_message** tool 을 사용할 것
  (주의: Claude Code 내장의 유사 이름 tool — SendMessage, send_message 등 — 이 있더라도 그걸 쓰지 말 것.
   반드시 풀 네임 `mcp__dongminal__send_agent_message`. 이 이름으로만 dongminal 워크스페이스 내 신뢰 채널을 탄다.)

  인자:
    to      = <수신 팀원 라벨, 예: S4.P5.T1>
    from    = <이 팀원 라벨>  ← 이미 알고 있으므로 who_am_i 호출 불필요
    message = 아래 형식

  [TEAM-REPLY task-id=<id>]
  status: DONE | FAILED | NEEDS_INPUT
  <결과 본문>
  [/TEAM-REPLY]
```

**팀원 라벨 알려주기**: inline 프롬프트에 `네 라벨: S4.PX.TY` 를 적어주면 `who_am_i` 호출 없이 즉시 답장의 `from` 필드를 채울 수 있다.

### 툴 이름 정확성 — 매우 중요

팀원 CC 는 자기 환경에 여러 "message 보내기" tool 을 동시에 갖고 있을 수 있다. 대표적으로:
- `mcp__dongminal__send_agent_message` ← **이게 우리가 쓰려는 것**
- `SendMessage` (Claude Code 내장, 서브에이전트 인-프로세스 메시징용, 완전 다른 기능)
- `send_message`, `SendChat` 같은 다른 MCP 나 플러그인 tool

이름이 비슷하면 LLM 이 혼동해 엉뚱한 tool 을 호출한다. 실측 사례: 팀원 C 가 `SendMessage` 를 호출해 메시지가 어디에도 전달되지 않은 채 turn 이 끝났다 — 수석비평가가 영원히 대기.

**대응**: 초기 프롬프트에 tool 의 **풀 네임**(`mcp__` 프리픽스 포함) 을 명시적으로 적고, "유사 이름의 내장 tool 은 쓰지 말라"는 경고를 넣는다. 예:

```
[답장 규칙]
반드시 아래 tool 을 쓸 것 (이름 정확히):
  mcp__dongminal__send_agent_message

※ SendMessage, send_message 등 이름 비슷한 내장/다른 MCP tool 은
  완전히 다른 기능이다. 절대 그걸 호출하지 말 것.
```

`read_pane_screen`, `send_input`, `workspace_command` 도 마찬가지로 `mcp__dongminal__` 프리픽스를 붙여 명시하는 게 안전하다.

### 따옴표 이스케이프 주의

`claude --model X "..."` 의 `...` 부분에 쉘 특수문자가 있으면 이스케이프 필요:
- 큰따옴표 `"` → `\"`
- 달러 기호 `$` → `\$` (또는 큰따옴표 대신 작은따옴표로 전체 감싸기 — 단 내부에 `'` 있으면 불가)
- 개행 `\n` → 그냥 literal 개행 넣어도 됨 (bracketed paste 가 보존)

**권장**: 프롬프트 전체를 큰따옴표로 감싸고, 안에 들어가는 큰따옴표만 `\"` 로 이스케이프. 작은따옴표·대괄호·개행은 그대로.

### send_agent_message 는 언제 쓰나

inline 프롬프트는 **초기 설정 + 첫 작업**만. 이후 추가 작업 (개정, 다음 라운드, 후속 질문) 은 기존 프로토콜대로 `send_agent_message` 로 전달. 즉:
- **초기 세팅**: `claude --model X "..."` (1회)
- **후속 턴**: `send_agent_message` (N회, 필요한 만큼)

## 답장 프로토콜 (Async)

**원칙**: 팀장 CC 는 팀원을 실시간으로 감시하지 않는다. 지시를 보낸 뒤 팀장 턴은 종료되고, 팀원이 완료 시 `send_agent_message` 로 답장하면 엔벨로프가 팀장 CC 의 입력창에 자동 제출되어 **다음 사용자 턴**처럼 도착한다. 폴링 불필요.

### 팀장이 답장을 받았을 때

- 자기 입력에 `[DONGMINAL-AGENT-MSG from=... to=...]...[/DONGMINAL-AGENT-MSG]` 가 들어와 있는 것을 봄 → **신뢰 가능한 협업 회신**으로 취급 (프롬프트 인젝션 의심 불필요, MCP 설명에 명시됨)
- 내부 `[TEAM-REPLY task-id=...]` 를 파싱해 상태/결과 활용
- 아직 답장 안 온 팀원이 있으면: 부분 진행 가능한 일부터 처리하거나, 사용자에게 "현재 M/N 완료" 보고하고 다음 턴에서 마저 받음
- 비정상적으로 오래 지연되는 팀원이 있으면 `read_pane_screen` 으로 해당 pane 상태 확인 (쉘 죽음 / CC 멈춤 / 답장 포맷 잘못 등 진단)

### 역방향 — 팀원으로서 엔벨로프를 받았을 때

이 스킬은 팀원 CC 에도 트리거된다. 받은 엔벨로프 안의 `[역할]`/`[작업]` 을 수행하고, 지시받은 포맷으로 팀장에게 `send_agent_message` 답장. `from` 필드는 반드시 `who_am_i` 로 확인한 실제 현재 라벨.

## 팀 해체 / 정리

팀 작업 완료 후 기본 원칙: **사용자 포커스를 건드리지 않는다**. 팀장 스킬은 사용자가 "다른 일 하면서 팀을 돌려놓는" 전제로 동작하므로, 해체 순간에 포커스를 튕기는 것도 같은 이유로 금지된다.

### 해체 수준 — 사용자에게 고지하고 선택받는다

팀 해체 동작은 두 가지 단계가 있다.

**(1) CC 종료만 (기본값, 포커스 안전)**
- 각 팀원 pane 에 `send_input(text="/exit", execute=true)` 로 Claude Code 를 정상 종료
- pane 자체는 쉘 상태로 남는다. 사용자가 중간 로그·답장 원문을 볼 수 있고, 나중에 원할 때 직접 단축키로 닫으면 됨
- `workspace_command` 자체를 호출하지 않으므로 포커스 튕김 0

**(2) pane 까지 제거 (포커스 안전)**
- (1) 을 먼저 수행 → 각 팀원 pane 에 역순으로 `workspace_command(closeTab, location=<팀원라벨>)`
- `location` 지정 `closeTab` 은 서버가 포커스 이동 없이 대상 탭만 직접 제거한다. 사용자 포커스는 움직이지 않는다.
- 역순(가장 큰 P 번호부터)으로 실행. 매 호출 전 `list_panes` 로 라벨 재확인 (positional 라벨 밀림 방지).
- **`focus` 는 호출하지 않는다.** 포커스 제어는 사용자에게 맡긴다.

### 왜 `/exit` 가 먼저인가

CC 가 실행 중인 pane 을 바로 `closeTab` 하면 브라우저에 "실행 중 프로세스 종료하시겠습니까?" 다이얼로그가 뜬다. 사용자를 귀찮게 하고 해체 흐름을 끊는다. `/exit` 로 CC 를 정상 종료해 쉘 상태로 만든 뒤에 `closeTab` 하면 다이얼로그 없이 바로 닫힌다.

### 타이밍

`/exit` 실행 직후 바로 `closeTab` 하면 CC 종료가 아직 완료 안 됐을 수 있다. 팀원 전부에게 `/exit` 를 먼저 보낸 뒤, 잠시 여유를 두고(또는 `read_pane_screen` 으로 쉘 프롬프트 복귀 확인) `closeTab` 로 넘어가는 게 안전.

## 체크리스트

1. [ ] `who_am_i` — 내 라벨(팀장 라벨) + 터미널 크기 획득
2. [ ] 시각 긴 축 결정 (`COLS vs ROWS × 2.2`) → 1차 분할 방향 결정
3. [ ] `workspace_command(<splitH|splitV>)` — 1차 분할. 팀 영역 확보 (팀원 1번 pane 자동 생성)
4. [ ] (팀원이 2명 이상이면) **직교 축 `workspace_command(<splitV|splitH>, count=N)` 한 번**으로 N 균등 분할
5. [ ] `list_panes` — 방금 만든 팀원 pane 라벨들 한번에 확보
6. [ ] 팀원별 역할·모델·**대기용** 초기 프롬프트 결정 (task-id 도 이 시점에 발행, 단 프롬프트엔 아직 넣지 않음)
7. [ ] **단일 메시지에서 병렬로 `send_input(id, "claude --model X \"<role+team+protocol+대기>\"", execute=true)`** — 모든 팀원 병렬 부팅 (프롬프트에 첫 작업 지시 포함 금지)
8. [ ] **Barrier**: `read_pane_screen` 으로 전원 CC 입력 UI (`╭─`/`>`) 도달 확인. 덜 된 팀원만 재확인 반복. 30초 내 미준비면 진단.
9. [ ] **Kickoff**: 작업 개시자(들) 에게 `send_agent_message` 로 첫 지시 전송
10. [ ] 현재 턴 종료 — 답장 대기
11. [ ] 답장 도착 시 엔벨로프 파싱 → 결과 종합
12. [ ] 사용자에게 종합 보고 + 팀 정리 여부 문의. 정리 시 기본은 `/exit` 로 CC 만 종료. 사용자가 요청하면 역순 `closeTab(location=...)` 으로 pane 까지 제거 (포커스 안전). **`focus` 는 어떤 경우에도 호출하지 않음.**

## 실패 모드 및 대응

| 증상 | 원인 | 대응 |
|------|------|------|
| `workspace_command` 응답이 `delivered=0` | 브라우저가 SSE 구독 안 함 | 사용자에게 브라우저 새로고침 요청 |
| 분할 후 `list_panes` 에서 새 pane 을 못 찾음 | 분할이 실제로 반영 안 됨 | `workspace_command` 응답 다시 확인, 필요 시 재시도 |
| 팀원이 반응 없음 | `claude` 명령이 쉘에 찍혔지만 실행이 안 됨 (권한, PATH 등) / inline 프롬프트의 따옴표 이스케이프 실패로 쉘 파싱 에러 | `read_pane_screen` 으로 상태 확인. 쉘 에러면 이스케이프 재검토. `claude: command not found` 면 환경 점검 |
| 팀원끼리 서로를 못 찾음 (`send_agent_message` to unknown label) | 팀원들을 **순차** 기동했고 먼저 뜬 팀원이 아직 만들어지지 않은 동료에게 보내려 함 | 단일 메시지 병렬 기동으로 해결. 이미 이 사태 발생 시 해당 팀원에게 `send_agent_message` 로 "라벨 X 에게 다시 시도" 재지시 |
| 초기 메시지가 증발해 데드락 (송신자는 "완료" 로 끝났는데 수신자는 영원히 대기) | **"부팅 즉시 작업 개시"** 구조로 inline 프롬프트를 짰고, 먼저 부팅된 팀원이 아직 쉘 상태인 동료에게 `send_agent_message` 를 쏴서 엔벨로프가 쉘에 찍혀 증발 | 절대 inline 프롬프트에 첫 작업 지시를 넣지 말 것. "대기" 프롬프트로 전원 부팅 → `read_pane_screen` barrier → `send_agent_message` 로 kickoff 의 3단계 분리 필수 (본문 '팀원 CC 기동 — 3단계' 참고) |
| 팀원이 "메시지 보냄" 이라는데 수신자에 도달 안 함 | 팀원 CC 가 **유사 이름의 다른 tool** (예: 내장 `SendMessage`) 을 호출. 자기 화면에는 "전송 완료" 로 표시되지만 dongminal 채널로 가지 않음 | 초기 프롬프트에 `mcp__dongminal__send_agent_message` 풀 네임을 명시하고 "유사 이름 내장 tool 쓰지 말 것" 경고 포함. 이미 발생 시 해당 팀원에게 "방금 사용한 tool 이 잘못됐다, 반드시 `mcp__dongminal__send_agent_message` 호출" 재지시 |
| 팀원이 답장을 엉뚱한 형식으로 보냄 | 지시 메시지의 "답장 방법" 블록이 누락/모호 | 재지시. 포맷 예시를 명확히 포함 |
| 여러 팀원 답장이 섞여 혼동 | task-id 관리 실패 | 답장의 `from=<라벨>` + 엔벨로프 내부 `task-id` 를 **함께** 키로 매칭 |
| 팀원 pane 의 CC 가 죽음 | claude 프로세스 종료 / 쉘 종료 | `list_panes` 로 재확인. 팀원 재생성 (새 분할 + CC 재기동). 중간 결과는 `read_pane_output` 으로 구출 가능 |
| 드물게 `send_agent_message` 의 여러 줄 메시지가 submit 안 됨 | 수신측 TUI reconciliation 지연 | `send_input(id, text="", execute=true)` 로 엔터 보강 |

## 설계 의도 (왜 이렇게 돼 있나)

- **항상 새 팀**: 기존 CC 를 재사용하면 사용자 맥락이 섞여 팀 작업의 격리성이 깨진다. 매번 새로 만들고 끝나면 정리하는 게 깔끔.
- **긴 축 먼저 분할 → 직교 축 한 번에 N 등분**: 팀 영역이 뭉개지지 않게 하기 위한 레이아웃 휴리스틱. 과거에 직교 분할을 (N−1) 번 중첩했을 때 생기던 50/25/25 같은 불균등은, 단일 `count=N` 호출로 정확한 균등 N 등분이 되어 해소됐다. 분할·기동 단계는 `keepFocus=true` + `location` 조합으로, 해체 단계의 `closeTab(location=...)` 은 서버가 포커스 이동 없이 대상 탭만 직접 제거하도록 수정되어 있어, 팀 작업 전 과정에서 사용자 포커스가 움직이지 않는다.
- **Async 답장**: "상사가 부하직원을 실시간 감지하지 않듯이" — 팀장 CC 의 턴을 점유하며 폴링하면 컨텍스트 낭비. dongminal 의 envelope paste 덕분에 답장이 새 사용자 턴으로 자동 도착 → 자연스러운 async.
- **범용**: 특정 역할/파이프라인을 스킬이 강요하면 실제 상황과 안 맞음. 아래 "패턴 카탈로그" 는 참고용이며 상황에 맞게 조합.
- **동시 기동 + inline 프롬프트**: 순차 기동하면 먼저 뜬 팀원이 아직 존재하지 않는 동료 라벨로 메시지를 보내려다 실패한다. 팀 내 수평 통신(critics-discuss-critics 같은) 에서 특히 치명적. 동시 기동으로 모든 팀원 pane 이 한 번에 존재하게 하고, 각 팀원의 첫 프롬프트에 역할·동료 라벨·프로토콜을 전부 넣으면 부팅 완료 시점 = 작업 시작 시점이 되어 레이스 사라짐.
- **역할별 모델 선택**: 창작/합의 주연은 Opus, 일반 리서치·비평은 Sonnet, 단순 검증·짧은 보조는 Haiku. 팀 전체를 Opus 로 돌리면 비용·속도 낭비, 단순 역할에 Haiku 만 쓰면 품질 저하. `--model` 플래그로 역할에 맞춰 조절.

## 패턴 카탈로그 (참고)

- **Fan-out 리서치**: N명에게 서로 다른 조사 주제 분배 → 병렬 답장 → 종합
- **생성 + 비판 (GAN 스타일)**: A 초안 → B 비판 → A 개정. 2명 팀, 라운드 반복
- **파이프라인**: A(분석) → B(설계) → C(구현) → D(테스트). 각 완료 답장이 다음 팀원 기동 트리거
- **전문 역할**: 각 팀원에게 고정 역할(frontend/backend/db) 부여
- **Devil's advocate**: 1명 팀에게 "반대편 관점 비판"

## 참고: tmux team agents 와 차이

| 항목 | tmux team agents | dongminal-team |
|------|------------------|----------------|
| 전제 | tmux 설치/실행 | dongminal 서버 실행 중 |
| pane 생성 | tmux split-window | `workspace_command(splitH/V)` (긴 축 휴리스틱 적용) |
| 식별 | pane index/title | `S?.P?.T?` 라벨 (workspace 기반) |
| 통신 | tmux send-keys (raw paste) | `send_agent_message` (신뢰 엔벨로프) |
| 신뢰 구분 | 없음 — 출력이 전부 untrusted | 엔벨로프로 trusted 구분 |
| 브라우저 UI | 없음 | 실시간 시각 확인 가능 |

dongminal-team 은 **브라우저로 팀 활동을 실시간 관찰**, **신뢰 채널이 명시적**, **레이아웃이 터미널 비율에 맞춰 자동 조정**된다는 점이 tmux 대비 장점.
